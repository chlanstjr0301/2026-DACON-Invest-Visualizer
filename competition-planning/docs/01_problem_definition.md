## 1. 문제 배경

### 1.1 왜 지금 이 문제인가

금융 기업과 투자 실무 현장에서는 포트폴리오 데이터를 Excel 또는 CSV 형식으로 수수하는 관행이 오래 유지되어 왔다. 거래내역 정산, 운용 성과 집계, 위험 지표 산출이 모두 이 파일 형식에서 출발하는 경우가 많다. 이 관행은 사라지지 않지만, 데이터를 다루는 도구 환경은 빠르게 변화하고 있다.

세 가지 변화가 이 문제를 가시화했다.

**첫째, AI 기반 데이터 분석 도구의 보급.** Julius AI, Power BI Copilot 등 일반 사용자가 CSV 파일을 업로드하고 자연어로 분석을 요청할 수 있는 도구가 시장에 안착했다. 이 도구들은 일반 비즈니스 데이터에 대해 편리한 UX를 제공하지만, 금융 포트폴리오 분석에 요구되는 도메인 규칙(수익률 계산, MDD 정의, 벤치마크 초과수익 기준 등)은 사용자가 직접 프롬프트로 정의해야 한다. 매번 재정의된 규칙은 세션 간 재현성을 보장하기 어렵다.

**둘째, 금융 특화 분석 서비스의 입력 구조 고정성.** Portfolio Visualizer, Sharesight, Koyfin과 같은 서비스는 성숙한 분석 기능을 제공하지만, 공개된 제품 설명을 검토하면 각각 티커 기반 입력, 브로커 연동, 또는 자체 포트폴리오 등록 방식을 주된 데이터 진입 경로로 제시한다. 사용자가 보유한 비정형 CSV의 컬럼명이 이들 시스템의 기대 스키마와 다를 경우, 직접 분석을 시작하기 위해서는 별도의 데이터 변환 작업이 필요하다.

**셋째, Skills.md 개념의 등장.** Anthropic의 Claude Code 에이전트 체계에서 Skills.md는 에이전트가 따를 분석 규칙과 행동 지침을 문서화한 명세서 역할을 한다. 이 개념을 금융 분석에 적용하면, 분석 규칙을 사전에 문서화하고 그 문서에 기반해 대시보드와 리포트를 자동 생성하는 구조가 가능해진다. 이 구조를 공개된 제품 사용 흐름에서 전면에 제시하는 서비스는 확인되지 않는다 (기획 해석).

이 세 가지 변화의 교차점에서, 비정형 투자 CSV/XLSX를 받아 금융 도메인 규칙에 따라 자동으로 포트폴리오 분석 대시보드와 애널리스트 형식의 리포트를 생성하는 서비스에 대한 공백이 명확해졌다.

### 1.2 문제의 핵심 구조

사용자는 두 가지를 가지고 있다. 데이터와 분석 목적이다.

| 사용자가 가진 것 | 현재의 장벽 |
|--------------|------------|
| 포트폴리오 CSV (컬럼 구조가 제각각) | 기존 금융 분석 서비스는 티커 입력·브로커 연동 등 특정 입력 스키마를 주된 진입 경로로 제시하는 경향이 있다 (기획 해석) |
| 분석 목적 (성과, 리스크, 비중 파악 등) | 범용 AI 도구는 금융 도메인 분석 규칙을 매번 사용자가 정의해야 한다 |

이 두 장벽이 겹치면, 사용자는 분석을 시작하기 위해 데이터 변환 작업을 직접 수행하거나, 도구마다 다른 방식으로 동일한 분석을 반복해야 한다.

## 2. 대상 사용자

MVP 단계에서 Skillfolio가 집중하는 사용자는 다음 두 페르소나다. **페르소나 A(투자 운용 실무자)가 P0 데모의 1차 타깃이며**, 페르소나 B(개인 퀀트 연구자)는 2차 타깃이다. Use Case와 P0 데모 시나리오는 페르소나 A의 월간 리포트 자동화 흐름을 대표 경로로 설계한다.

### 페르소나 A: 투자 운용 실무자

| 항목 | 내용 |
|------|------|
| 역할 | 투자 리포트 작성자, 포트폴리오 분석 담당자, 운용 성과를 정리하는 금융 데이터 실무자 |
| 컨텍스트 | 월간 또는 분기별로 운용 성과 보고서를 작성해야 한다. 데이터는 주로 사내 시스템에서 CSV로 내려받거나 거래 상대방에게서 Excel로 수수한다 |
| 보유 데이터 | 일별 포트폴리오 수익률, 보유 종목별 비중, 거래 내역, 자산군 분류 정보 |
| 필요한 분석 | 누적 수익률, 벤치마크 대비 초과수익, 최대낙폭(MDD), 변동성, 비중 분포, 집중도 위험 |
| 현재 사용 도구 | Excel, Google Sheets, 사내 보고 시스템, 또는 Python |
| 도구의 한계 | Excel은 분석 로직을 매번 수작업으로 재구성해야 하며, 같은 수식이 파일마다 다르게 정의되어 재현성이 떨어진다. Python은 유연하지만 시각화와 리포트 정리까지 추가 작업이 필요하다. 사내 보고 시스템은 정해진 양식에 맞춰 사용해야 해서 포트폴리오 분석 흐름을 자유롭게 구성하기 어렵다 |

### 페르소나 B: 개인 퀀트 연구자 / 투자 분석 스터디 참여자

| 항목 | 내용 |
|------|------|
| 역할 | 퀀트 입문자, 투자 분석 공부 목적의 개인 투자자, 금융 데이터 분석 과제를 수행하는 학생 |
| 컨텍스트 | 공개 데이터나 직접 수집한 가격·수익률 데이터를 분석하고 그 결과를 시각화해 공유하려 한다 |
| 보유 데이터 | 다운로드한 주가 CSV, ETF 수익률 데이터, 직접 구성한 종목별 비중 시트 |
| 필요한 분석 | 성과 분해, 드로다운 분석, 섹터별 비중 검토, 기초 리스크 지표 |
| 현재 사용 도구 | Python(pandas, matplotlib), Excel, Julius AI 등 범용 AI 분석 도구 |
| 도구의 한계 | Python은 분석 코드를 직접 작성해야 하며 시각화 결과를 리포트 형식으로 정리하는 데 추가 작업이 필요하다. 범용 AI 도구는 금융 지표 정의를 매번 프롬프트에 포함해야 하며, 동일한 분석을 재현하기 어렵다 |

## 3. 사용자 Pain Point

| # | Pain Point | 설명 | 해당 페르소나 |
|---|-----------|------|-------------|
| P1 | 파일을 올려도 분석이 바로 시작되지 않는다 | 기존 금융 분석 서비스에서 요구하는 스키마와 사용자의 CSV 컬럼명이 다르면, 매핑 작업을 사용자가 직접 수행해야 한다. 이 준비 작업이 분석보다 더 긴 경우가 많다 | A, B |
| P2 | 같은 데이터를 넣어도 분석 결과가 달라진다 | 범용 AI 분석 도구는 자연어 프롬프트에 따라 결과가 달라질 수 있다. 성과 보고서나 리스크 검토 목적의 분석에서 결과 재현성이 보장되지 않으면 업무 신뢰성이 낮아진다 | A |
| P3 | 데이터가 부족할 때 시스템이 오류를 내거나 침묵한다 | 요청한 분석에 필요한 컬럼이 없거나 데이터 기간이 짧을 때, 기존 도구는 에러를 출력하거나 아무 안내 없이 분석을 건너뛴다. 어떤 분석이 가능하고 어떤 추가 데이터가 필요한지 안내받지 못한다 | A, B |
| P4 | 차트를 만들었지만 왜 그런 결과가 나왔는지 설명하기 어렵다 | 범용 AI 도구나 시각화 툴로 차트를 생성했더라도, 어떤 계산 규칙을 사용했는지, 어떤 기준으로 인사이트를 도출했는지 추적할 수 없어 보고서에 첨부하기 어렵다 | A, B |
| P5 | 분석 결과를 바로 보고서로 쓸 수 없다 | 차트와 수치를 따로 만든 뒤 Word 또는 PowerPoint에 붙여넣는 수작업이 필요하다. 대시보드와 리포트가 분리된 워크플로는 반복 작업을 유발한다 | A |
| P6 | 데이터 형식마다 분석 흐름을 다시 설계해야 한다 | 일별 수익률 파일과 보유 종목 비중 파일은 형식이 다르다. 같은 분석 목표여도 데이터 구조가 바뀌면 전체 분석 흐름을 새로 구성해야 한다 | A, B |

## 4. 기존 서비스의 한계

> **면책 표기:** 이하의 서비스별 한계 서술은 각 서비스의 공개된 제품 설명, 공식 문서, 웹사이트 기능 소개를 기반으로 한 기획 해석이다. 각 서비스가 공식적으로 자인한 결함이 아니다.

| 서비스 | 서비스 유형 | 공개 정보 기반 특징 | 우리의 기획 해석 (한계) |
|--------|-----------|------------------|----------------------|
| Portfolio Visualizer | 금융 특화 포트폴리오 분석 | 백테스트, 최적화, 리스크 분석을 지원하며 분석은 자체 데이터베이스의 티커/심볼 기반으로 수행됨 | 사용자가 보유한 임의의 CSV 파일을 직접 업로드해 분석하는 흐름은 주된 사용 경로로 제시되지 않음. 비정형 컬럼 구조의 파일은 별도 변환 없이 바로 사용하기 어려운 구조로 파악됨 |
| Koyfin | 금융 리서치 / 포트폴리오 플랫폼 | 포트폴리오 모니터링, 성과 추적, 맞춤형 대시보드를 제공하며, holdings는 Koyfin 자체 시스템에 등록하거나 브로커 연동으로 입력 | 사용자 고유의 비정형 CSV를 직접 분석 단위로 사용하는 시나리오가 주된 기능으로 문서화되어 있지 않음. 시장 데이터 라이선스와 커버리지 범위가 넓어 해커톤 MVP 수준의 재현은 어려움 |
| Sharesight | 포트폴리오 트래커 / 성과 리포팅 | 200개 이상 브로커 연동과 70만 종목 추적을 지원하며 거래내역 기반 성과, 배당, 세금 리포트를 생성 | 브로커 연동 또는 이메일 자동 가져오기가 주된 데이터 진입 경로. 임의의 CSV 업로드로 비정형 컬럼을 처리하는 유연한 매핑 기능은 공식 문서에서 확인되지 않음 |
| Power BI Copilot | 범용 BI + AI 리포트 생성 | CSV/Excel 업로드를 지원하고 자연어 프롬프트로 대시보드와 리포트를 생성함. Microsoft 공식 문서는 Copilot 생성 보고서가 비결정적(nondeterministic)이며 도메인 전문가를 대체할 수 없다고 명시 | Skillfolio가 제안하는 Skills.md 기반 규칙 정의 구조, 포트폴리오 특화 지표 명세, 데이터 충분성 검증 워크플로는 Power BI Copilot이 전면에 내세우는 방식과 다른 설계 선택이다. Microsoft 공식 문서가 명시한 비결정성은 이 차이를 뒷받침하는 참고 근거로 활용한다 (기획 해석) |
| Julius AI | 범용 AI 데이터 분석 | CSV, Excel, Google Sheets를 업로드해 자연어로 분석, 차트, 통계 모델링을 수행하는 범용 도구 | Skillfolio가 제안하는 Skills.md 기반의 규칙 명세, 포트폴리오 목적 기반 분석 라우팅, 데이터 충분성 검증 워크플로를 전면에 내세우는 구조가 공개된 제품 기능 설명에서 확인되지 않음 (기획 해석) |

## 5. 왜 이 문제가 공모전 주제와 맞는가

이 공모전의 공식 명칭은 "월간 해커톤: 투자 데이터를 시각화하라 — Skills 기반 대시보드 설계"다. 평가 배점은 범용성(25), Skills.md 설계(25), 대시보드 자동 생성(25), 바이브코딩 활용(15), 실용성 및 창의성(10)으로 구성된다.

우리가 정의한 문제는 이 평가 구조와 다음과 같이 직접 대응한다.

| 공모전 평가항목 | 배점 | 문제 정의와의 연결 |
|--------------|-----|-----------------|
| 범용성 | 25 | 포트폴리오 수준의 투자 데이터를 대상으로 하되, 컬럼 구조가 다른 CSV/XLSX를 AI 매핑으로 처리하는 설계는 "재사용 가능하고 다양한 데이터 구조에 대응하는" 범용성을 금융 도메인 안에서 구현한다 |
| Skills.md 설계 | 25 | P2(분석 재현성 부재), P4(분석 근거 불투명)는 분석 규칙이 문서화되지 않았기 때문에 발생하는 문제다. Skills.md가 이 규칙을 명세하고 결과와 연결됨을 보이는 것이 문제 해결의 핵심이다 |
| 대시보드 자동 생성 | 25 | P1(매핑 수작업), P5(보고서 별도 조합), P6(데이터 형식마다 재설계)는 대시보드 자동 생성이 해결하는 마찰 지점이다. 업로드에서 대시보드 생성까지의 자동화 흐름이 이 항목과 직결된다 |
| 바이브코딩 활용 | 15 | Skills.md를 코드 생성의 기준 문서로 활용하고, 그 문서에서 대시보드 컴포넌트가 생성되는 구조를 갖추면 "문서 기반 생성 구조" 기준을 충족한다 |
| 실용성 및 창의성 | 10 | P3(데이터 부족 시 무응답), P5(보고서 수작업)는 실무 사용자가 매일 직면하는 마찰이다. 이 마찰을 줄이는 서비스는 실제 활용 가능성 기준에서 긍정적으로 평가될 수 있다 |

## 6. 문제 범위

Skillfolio가 해결하는 문제는 다음 범위로 한정된다.

### 6.1 입력 데이터 범위

| 지원 항목 | 세부 내용 |
|---------|---------|
| 파일 형식 | CSV, XLSX |
| 데이터 유형 | 포트폴리오 보유 종목(holdings), 일별 수익률, 일별 자산 가격 (P0 핵심); 단순 거래 내역 (P1 — MVP에서는 거래 목록 인식에 한정, 성과 계산에 사용하지 않음) |
| 데이터 주기 | 일별(daily) 기준 |
| 최소 요건 | 최소 2개 이상의 종목 또는 포트폴리오 전체를 포함하는 데이터 |

### 6.2 분석 범위

| 분석 항목 | 필요 데이터 |
|---------|-----------|
| 성과 분석 | 일별 수익률 또는 일별 가격 |
| 드로다운 분석 | 일별 수익률 또는 누적 수익률 시계열 |
| 리스크 분석 | 일별 수익률 (연환산 변동성 P0, 샤프 지수 P1) |
| 비중 분석 | 보유 종목 비중 또는 평가금액 정보 |
| 집중도 분석 (P1) | 보유 종목별 비중 정보 |
| 벤치마크 비교 (P1) | 벤치마크 데이터가 파일에 포함된 경우에 한정 |

### 6.3 가정 및 제약

- 사용자가 업로드하는 파일은 하나 이상이며, 파일 간 결합은 공통 날짜 또는 종목명 컬럼을 기준으로 한다.
- 외부 시장 데이터 API 연결은 MVP 범위에 포함하지 않는다.
- 벤치마크 비교는 사용자가 벤치마크 수익률 또는 가격 데이터를 동일 파일 또는 별도 파일로 제공한 경우에만 수행한다.
- 분석 결과의 정확성은 사용자가 제공한 데이터의 품질에 의존한다. 시스템은 데이터 검증 결과를 사용자에게 표시하되, 원본 데이터를 임의로 수정하지 않는다.

## 7. 비문제 / 제외 범위

다음 항목은 Skillfolio MVP가 해결하는 문제의 범위 밖이다. 이 항목들은 구현 난이도, MVP 집중도, 공모전 핵심 평가 항목과의 정렬을 고려해 명시적으로 제외한다.

| 제외 항목 | 제외 이유 |
|---------|---------|
| 실시간 시장 데이터 | 외부 API 연동 없이 사용자 업로드 파일만 분석하는 것이 Skillfolio의 설계 원칙이다. 실시간 데이터는 라이선스, 레이턴시, 심사 기간 중 키 노출 문제를 유발한다 |
| 자동 거래 및 주문 실행 | 투자 자동화는 금융 규제, 브로커 연동, 리스크 고지 의무를 수반한다. Skillfolio는 분석과 리포트 생성에서 멈춘다 |
| 세금 계산 및 세금 리포트 | 국가별 세법 적용이 필요한 세금 계산은 법적 책임과 도메인 복잡도가 크다. 성과·리스크·비중·집중도 분석으로 범위를 한정한다 |
| 틱 데이터 및 분봉 이하 마이크로스트럭처 분석 | 고빈도 데이터 처리와 시장 미시구조 분석은 포트폴리오 수준 분석과 목적이 다르다. 일별 데이터 기준으로 범위를 제한한다 |
| 팩터 리서치 플랫폼 기능 | Fama-French 팩터 회귀, 스타일 분석 등 고급 팩터 분석은 MVP 범위를 초과한다. 성과와 리스크의 기초 지표에 집중한다 |
| 전사적 BI 플랫폼 | Power BI나 Tableau 수준의 범용 데이터 연결, 대용량 데이터 파이프라인, 조직 권한 관리는 이 서비스의 목적이 아니다 |
| 브로커 API 연동 | 브로커 API를 통한 자동 데이터 수집은 인증, 보안, 브로커별 스키마 차이 문제가 있다. 파일 기반 업로드만 지원한다 |
| 단일 종목 분석 | 개별 종목의 재무 분석, 밸류에이션, 실적 예측은 포트폴리오 수준 분석과 목적이 다르다. 최소 포트폴리오 단위를 분석 대상으로 한다 |
