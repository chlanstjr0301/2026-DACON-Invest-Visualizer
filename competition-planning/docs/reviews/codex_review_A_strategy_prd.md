## 1. Overall verdict

현재 문서의 핵심 방향은 좋다. Skillfolio는 "범용 BI"도 "범용 챗봇"도 아니라, **비정형 투자 CSV/XLSX를 Skills.md 규칙으로 정규화하고 재현 가능한 포트폴리오 대시보드와 리포트로 변환하는 서비스**라는 포지션을 비교적 일관되게 유지하고 있다. 공모전의 75점 핵심 배점인 범용성, Skills.md 설계, 대시보드 자동 생성과도 직접 연결된다.

다만 Stage 4 Use Case로 넘어가기 전에는 P0 범위를 더 단단히 잠가야 한다. 특히 거래내역 지원, AI 애널리스트 패널, Skills 로그, 재현성 정의, 리포트 다운로드, 집중도/샤프/벤치마크 관련 우선순위가 문서마다 조금씩 다르게 읽힌다. 이 상태로 Use Case를 쓰면 사용 흐름이 과도하게 커지고, 이후 화면/기능 명세가 MVP보다 큰 제품으로 번질 가능성이 높다.

판정은 **조건부 Go**다. 큰 방향은 유지하되, Use Case 작성 전에 "P0 데모 경로"와 "P1 이후 기능"을 더 엄격하게 분리해야 한다.

## 2. Critical issues

### 2.1 P0 scope creep가 아직 남아 있다

PRD는 일부 항목을 P1/P2로 내렸지만, 시나리오와 기능 설명에는 여전히 P0처럼 읽히는 항목이 남아 있다.

대표 사례:
- PRD 섹션 4.3에서는 집중도 분석을 P1로 분류하지만, 시나리오 1에서는 "집중도 분석이 가능하다"고 정상 흐름에 포함된다.
- Markdown 파일 내보내기는 PRD 섹션 4.4에서 P1인데, 시나리오 1에서는 사용자가 "리포트 다운로드"를 클릭한다고 되어 있다.
- 벤치마크 비교는 P1이고 파일 내 데이터가 있을 때만 가능하지만, 시나리오 4에서는 추가 업로드 후 대시보드 갱신까지 포함되어 있어 P0 제품 흐름처럼 보인다.

수정 방향:
- P0 데모 경로는 "5종목 1년치 일별 수익률 + 선택적 weight 컬럼" 하나로 고정하는 것이 좋다.
- P0 분석은 누적 수익률, 기간 수익률, MDD, 드로다운 차트, 연환산 변동성, 종목별 비중 정도로 제한한다.
- 집중도, 샤프 지수, 벤치마크 비교, Markdown 다운로드, 다중 파일 결합은 P1로 명확히 격리한다.

### 2.2 거래내역 지원 범위가 너무 넓게 보인다

문서 전반에서 "simple transaction history"가 지원 데이터 유형으로 반복 등장한다. PRD에서는 "거래 목록 표시 및 보유 내역 요약에 한정"한다고 좁혔지만, 문제 정의와 제품 개념에서는 거래내역이 성과 분석의 자연스러운 입력처럼 읽힐 여지가 있다.

거래내역은 생각보다 복잡하다. 매수/매도, 수수료, 분할매수, 현금흐름, 배당, 매도 실현손익, 미실현손익, 평가 기준일, 가격 시계열 결합이 모두 걸린다. "단순 거래 내역"이라는 이름만으로도 심사위원은 Sharesight류 기능을 기대할 수 있다.

수정 방향:
- P0에서는 거래내역을 지원 입력 유형에서 제외하거나, "업로드 및 컬럼 인식 데모만 가능, 성과 계산에는 사용하지 않음"으로 더 강하게 제한한다.
- 더 나은 선택은 P0 지원 데이터 유형을 `daily returns`, `daily asset prices`, `holdings` 3개로 줄이는 것이다.
- 거래내역은 P1 또는 확장 로드맵에서 "holdings 재구성 및 현금흐름 기반 성과 계산"으로 별도 정의해야 한다.

### 2.3 AI 애널리스트 패널이 아직 자유서술 의존으로 보일 수 있다

문서가 AI 텍스트의 비결정성을 명시한 점은 좋다. 그러나 AI 애널리스트 패널이 P0로 들어가면서 "지표 해석, 위험 요인, 데이터 한계 고지"까지 담당한다. 이 기능이 자유서술 중심이면 제품의 핵심 주장인 재현성과 충돌한다.

수정 방향:
- AI 애널리스트 패널의 P0 역할은 "자유로운 분석"이 아니라 "구조화된 템플릿 채우기"로 정의해야 한다.
- P0 리포트 문장은 다음처럼 제한하는 것이 좋다.
  - 입력: 결정적 지표값, 데이터 충분성 결과, Skills.md 규칙 ID
  - 출력 섹션: 요약, 관찰된 수치, 위험 경고, 데이터 한계
  - 금지: 투자 권유, 원인 추정 과잉, 미래 전망, 데이터에 없는 외부 시장 설명
- "AI 보조 생성" 표기만으로는 부족하다. 어떤 문장은 템플릿 기반이고, 어떤 문장만 LLM 보조인지 구분해야 한다.

### 2.4 재현성 정의는 방향은 맞지만 더 정밀해야 한다

문서는 결정적 출력과 비결정적 출력을 분리한다. 이는 강점이다. 다만 "대시보드 레이아웃 구성"과 "차트 유형 선택"까지 결정적이라고 주장하려면 입력 조건과 규칙 우선순위가 더 구체적이어야 한다.

현재 필요한 추가 정의:
- 동일 입력 파일의 판정 기준: 파일 내용 해시인지, 컬럼 순서까지 포함하는지, 시트 선택까지 포함하는지
- 동일 매핑의 판정 기준: 사용자 확정 매핑만 기준인지, AI 신뢰도 점수도 로그에 포함하는지
- 동일 Skills.md 규칙 버전의 판정 기준: 규칙 ID별 버전인지, 전체 Skills.md 파일 버전인지
- 차트 선택 충돌 시 우선순위: 수익률과 가격이 모두 있을 때 어느 컬럼을 우선하는지
- 데이터 정렬/결측/중복 처리 규칙: 재현성에 직접 영향을 준다

수정 방향:
- 재현성 문구를 "전체 서비스 결과가 재현된다"가 아니라 "확정된 매핑 이후의 결정적 계산과 규칙 기반 레이아웃 선택이 재현된다"로 통일한다.
- Stage 4 전에 "Reproducibility Contract" 표를 하나 추가하는 것이 좋다.

### 2.5 Skills 로그가 핵심 기능이지만 아직 산출물 수준이 모호하다

Skills 로그는 이 기획의 차별화 포인트다. 그러나 현재는 "규칙 ID, 버전, 입력 컬럼, 결과값 기록" 수준이다. 심사위원이 실제로 보고 납득하려면 로그가 단순 감사 기록이 아니라 **Skills.md가 실제로 대시보드를 제어했다는 증거**여야 한다.

수정 방향:
- 각 대시보드 카드에 "참조 규칙 ID"가 붙어야 한다.
- 로그는 최소한 다음 단위로 나누어야 한다.
  - Column Mapping Log
  - Sufficiency Check Log
  - Metric Calculation Log
  - Visualization Rule Log
  - Insight Rule Log
- "AI가 작성한 문장"도 참조 수치와 금지된 추론 여부를 기록해야 한다.
- Use Case에는 사용자가 Skills 로그를 열어 "왜 이 차트가 생성됐는지" 확인하는 흐름을 포함해야 한다.

### 2.6 경쟁 서비스 한계 서술은 조심스럽지만, 일부는 여전히 과장으로 보일 수 있다

문서가 여러 번 "기획 해석"이라고 면책한 점은 좋다. 다만 "기존 도구 어느 곳도 이 구조를 정면으로 구현하고 있지 않다", "전체 범주 공통으로 분석 규칙 문서가 확인되지 않는다" 같은 표현은 방어적 근거가 부족하면 과장으로 보일 수 있다.

수정 방향:
- 경쟁사 한계는 "없다"보다 "공개 제품 설명의 주된 사용 흐름에서는 전면에 드러나지 않는다"로 낮춰 표현한다.
- "우리가 유일하다"가 아니라 "공모전 맥락에서 제출물로 보여주기 좋은 조합이다"로 전환하는 것이 안전하다.
- 경쟁 서비스 대비 차별점은 기능 우월성보다 "Skills.md 제출물과 대시보드 동작의 연결을 명시적으로 보여준다"에 집중해야 한다.

## 3. Important but non-blocking issues

### 3.1 문제 정의는 구체적이지만 대상 사용자가 둘로 갈라진다

투자 운용 실무자와 개인 퀀트 연구자는 모두 타당하지만, MVP 메시지는 실무자 중심이 더 강하다. 개인 퀀트 연구자를 같은 비중으로 두면 UX와 데이터 샘플이 학습용 도구처럼 보일 수 있다.

권장:
- 1차 타깃: 투자 운용/리포트 실무자
- 2차 타깃: 개인 퀀트 연구자
- Use Case도 페르소나 A의 월간 리포트 자동화 흐름을 대표 시나리오로 둔다.

### 3.2 Sharpe ratio는 P1로 두는 것이 맞다

PRD에서 샤프 지수를 P1로 내린 것은 적절하다. 무위험 수익률 가정, 기간 짧음, 연환산 방식, 음수 수익률 해석 때문에 설명 비용이 크다. P0에서는 연환산 변동성과 MDD만으로도 리스크 분석 데모가 충분하다.

주의:
- 문제 정의와 제품 개념에는 아직 "샤프 지수 등"이 MVP 분석처럼 보이는 표현이 있다. 모두 "P1 확장"으로 정리해야 한다.

### 3.3 Treemap은 P1로 두는 것이 맞다

트리맵은 시각적으로 좋지만 P0 필수는 아니다. 비중 분석은 파이차트 또는 막대차트로 충분하다. 트리맵을 P0에 넣으면 화면 명세와 반응형 UI 부담이 커진다.

권장:
- P0: 종목별 비중 막대차트 또는 파이차트
- P1: 트리맵, 자산군/업종 drill-down

### 3.4 PRD는 구현-heavy 직전까지 갔지만 아직 기획 문서로는 허용 가능하다

PRD는 Feature ID, 수용 기준, 엣지 케이스까지 포함해 다소 상세하다. 하지만 공모전 제출물은 구현 기획서와 Skills.md 설계를 요구하므로 이 정도 상세함은 장점이 될 수 있다.

다만 이후 문서에서는 코드/프레임워크/API 세부로 들어가지 말고, "심사위원이 기능 동작을 확인할 수 있는 기준"에 머물러야 한다.

### 3.5 범용성 점수와 MVP 축소 사이의 긴장이 있다

범용성 25점을 의식해 holdings, returns, prices, transactions 4종을 모두 잡았지만, MVP 신뢰도는 오히려 낮아질 수 있다. 범용성은 데이터 유형 수보다 "스키마가 달라도 같은 분석 목적에 도달하는 능력"으로 증명하는 편이 낫다.

권장:
- P0 데모 데이터셋 2개만 준비한다.
  - Dataset A: 일별 수익률 중심
  - Dataset B: holdings + 평가금액 중심
- 같은 Skills.md 규칙이 두 데이터셋에서 어떻게 다른 대시보드를 생성하는지 보여준다.

## 4. Suggested revisions by file

### 4.1 `00_strategy_memo.md`

- MVP 데모 시나리오의 "PDF 또는 PNG 내보내기"를 제거하거나 P2로 명시한다.
- "재현 가능" 표현을 "확정 매핑 이후의 수치 계산 및 규칙 기반 대시보드 구성 재현 가능"으로 좁힌다.
- 경쟁 서비스 한계는 "부재"가 아니라 "공개 사용 흐름에서 전면화되지 않음"으로 낮춘다.
- 대중 투표 전략에는 "한 개의 강한 데모" 원칙을 추가한다. 기능 수보다 3초 내 이해 가능한 결과 화면이 중요하다.

### 4.2 `01_problem_definition.md`

- 대상 사용자는 페르소나 A를 1차 타깃으로 명시하고, 페르소나 B는 보조 타깃으로 낮춘다.
- 입력 데이터 범위에서 "단순 거래 내역"을 P1로 이동하거나, P0에서는 성과 계산에 사용하지 않는다고 명시한다.
- 리스크 분석 필요 데이터의 "샤프 지수 등" 표현을 "연환산 변동성 중심, 샤프 지수는 확장"으로 수정한다.
- 문제 정의의 결론에 "단일 종목 분석이 아니라 최소 포트폴리오 단위"라는 제약을 더 앞쪽에 배치한다.

### 4.3 `02_product_concept.md`

- 핵심 가치 제안 V2에서 재현성 범위를 더 짧고 엄격하게 표현한다.
- 서비스 흐름 7단계의 "저장"은 화면 내 Markdown 확인으로 제한하고, 파일 다운로드는 P1/P2로 분리한다.
- MVP 지원 데이터 유형에서 거래내역을 제외하거나 "인식만 가능"으로 바꾼다.
- 핵심 분석 항목에서 샤프 지수, 집중도, 벤치마크 비교의 우선순위를 PRD와 맞춘다.
- "애널리스트 리포트"는 자유 생성이 아니라 "Skills.md 기반 리포트 템플릿 + AI 보조 문장"으로 재정의한다.

### 4.4 `03_prd.md`

- 시나리오 1에서 집중도 분석 가능, 리포트 다운로드를 제거하거나 P1 플래그를 붙인다.
- 시나리오 4는 Stage 4 Use Case 후보에서 P1로 분류한다. P0 Use Case에 넣으면 범위가 커진다.
- F-UP-03 다중 파일 업로드가 P0라면 벤치마크 비교 P1과 충돌한다. P0에서는 단일 파일 업로드로 제한하거나, 다중 파일은 "동일 유형 파일 결합 없음, 추가 업로드만 가능"처럼 축소한다.
- F-AN-05 집중도 분석은 우선순위가 누락되었거나 P1로 보이는데, 수용 기준은 P0처럼 상세하다. 명확히 P1로 표시한다.
- F-AN-06 벤치마크 비교도 우선순위를 P1로 명시하고, P0에서는 "데이터 부족 안내 패널"까지만 제공한다고 분리한다.
- 섹션 7에서 샤프 지수는 "Skills.md 기본값 사용 후 계산"이라고 되어 있어 P1이 아니라 P0처럼 읽힌다. P0에서는 계산하지 않고 "P1: 무위험 수익률 가정 필요"로 정리한다.
- Skills 로그 포맷에 `rule_input`, `rule_output`, `calculation_formula`, `chart_selection_reason`, `excluded_analysis_reason`을 추가하면 공모전 설득력이 커진다.

## 5. Questions for the project owner

1. P0 데모에서 정말 거래내역 파일을 지원해야 하는가? 아니면 거래내역은 P1로 미루고 일별 수익률/가격/holdings에 집중할 것인가?
2. P0에서 다중 파일 업로드가 필수인가? 벤치마크 비교를 P1로 둔다면 단일 파일 중심이 더 일관적이다.
3. AI 애널리스트 패널은 자유 생성 리포트인가, 아니면 Skills.md 템플릿에 수치와 경고를 채우는 구조인가?
4. Skills 로그를 사용자에게 어느 수준까지 노출할 것인가? 카드별 규칙 ID만 보여줄 것인지, 계산식과 입력 컬럼까지 보여줄 것인지 결정이 필요하다.
5. 재현성의 기준은 세션 단위인가, 파일 해시 + Skills.md 버전 단위인가?
6. P0의 대표 사용자는 투자 운용 실무자인가, 개인 퀀트 연구자인가? 두 사용자 모두를 같은 무게로 두면 Use Case가 분산된다.
7. 공모전 제출 웹 페이지에서 "바이브코딩 활용"을 어떻게 증명할 것인가? 단순 설명이 아니라 Skills.md → 생성된 대시보드의 연결 증거가 필요하다.

## 6. Go / No-Go recommendation for Stage 4

**조건부 Go.**

Stage 4 Use Case로 넘어가도 되지만, 먼저 아래 네 가지를 정리해야 한다.

1. P0 데모 경로를 하나로 고정한다.
   - 권장: "5종목 1년치 일별 수익률 CSV 업로드 → 컬럼 매핑 확인 → 충분성 검사 → 성과/MDD/변동성/비중 대시보드 → Skills 로그 확인"

2. P0에서 제외할 기능을 명확히 선언한다.
   - 거래내역 기반 성과 계산
   - 샤프 지수
   - 집중도/HHI
   - 벤치마크 비교
   - 트리맵
   - Markdown 파일 다운로드
   - PDF/PNG 내보내기
   - 다중 파일 결합

3. AI 애널리스트 패널을 템플릿 기반으로 제한한다.
   - 자유로운 투자 분석이 아니라, 결정적 지표와 Skills.md 규칙을 기반으로 한 구조화된 설명이어야 한다.

4. Skills 로그를 Stage 4 Use Case의 핵심 장면으로 포함한다.
   - 단순 부가 패널이 아니라 "Skills.md가 실제로 작동했다"는 공모전 증거물로 설계해야 한다.

위 네 가지가 반영되면 Stage 4는 Go다. 반영하지 않으면 Use Case 문서가 너무 넓어지고, 이후 User Flow와 Screen Spec에서 MVP 실현 가능성이 급격히 약해질 가능성이 높다.
